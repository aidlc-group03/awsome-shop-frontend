# Full e-commerce feature implementation: services, mock, pages, and Docker

This change adds the complete feature set for an employee points-redemption shop and its admin back-office. It introduces a typed domain model, a dual-mode service layer (mock vs real API), six admin CRUD pages, seven employee-facing pages (product browsing, redemption flow, orders, points center), shared UI components, i18n expansion, and a production Docker setup. The auth store is refactored to delegate to the service layer, and the request interceptor now unwraps `ApiResult<T>` envelopes.

Watch for: missing i18n keys causing visible raw key strings in the employee shop (confirmed); mock session state lost on page refresh silently falling back to hardcoded user (confirmed); client-side status=1 filtering that corrupts pagination totals (confirmed); double-negative sign display for spend transactions in PointsCenter (confirmed); 401 handler leaving stale Zustand auth state (confirmed).

## High-level view

The service layer uses a `VITE_USE_MOCK` environment variable evaluated at call time to choose between a local in-memory mock and real axios calls. The mock's reliance on module-level `currentUserId` means page refresh loses session identity, causing user-scoped endpoints to silently return wrong data.

The request interceptor detects `ApiResult` shapes by checking for a `code` property on the response body and unwraps `.data` on success. The 401 handler clears localStorage token and hard-navigates but doesn't clear Zustand's persisted auth state, creating a flash-of-authenticated-UI on next load.

The employee pages implement business rules BR-F2 through BR-F5 with one structural gap: `ShopHome` filters `status=1` products client-side after receiving a page from the API, then still uses the server's `pages` count for pagination, producing incorrect page totals when delisted products exist.

i18n keys `employee.allCategories` and `employee.noProducts` are referenced in `ShopHome` but never defined in either locale file. The `ProductCard` component hardcodes the Chinese string "已售罄" rather than using an i18n key, violating BR-F9.3. The `ConfirmDialog` and `EmptyState` components also use Chinese default props.

The PointsCenter transaction list displays a type-based prefix ("+"/"-") concatenated with the raw `tx.points` value, which is already negative in the mock data for spend/deduct types. This produces `--2,580` in the UI.

## Service layer type safety at the boundary

Every service function casts the axios return via `as Promise<T>` without runtime validation. The interceptor strips the axios response envelope, returning the unwrapped payload directly. When the real backend is connected and the response shape deviates (extra wrapping, different field names, null where undefined is expected), TypeScript won't catch it — the cast papers over the gap. This is a known trade-off for this project stage but will surface as runtime crashes during integration.

## 401 / session lifecycle gap

When a 401 arrives, the interceptor does:
```ts
localStorage.removeItem('token');
window.location.href = '/login';
```

The Zustand store persists `{ user, token, isAuthenticated }` under `auth-storage` in localStorage. Removing `token` alone leaves `isAuthenticated: true` and `user` intact. On next page load the `AuthGuard` reads Zustand's persisted state, sees `isAuthenticated: true`, and renders the protected layout. The first API call then fails (no token in the separate `token` localStorage key), triggering another 401 cycle. The fix: also remove the `auth-storage` key, or import and call the store's `logout()` method from the interceptor.

## Client-side status filtering in ShopHome

```ts
const result = await productService.getList(params);
const activeProducts = result.records.filter((p) => p.status === 1);
setProducts(activeProducts);
setTotalPages(result.pages);  // ← stale, doesn't account for filtered-out items
```

If the API returns a page with 20 products where 3 are `status=0`, the grid shows 17 items but pagination indicates the full page count. The mock's `mockGetProductList` doesn't filter by status, so any product toggled to status=0 in the admin panel remains in the employee shop's response. The fix: add a `status` filter parameter to `mockGetProductList` and pass `status: 1` from `ShopHome`, or adjust `totalPages` based on the post-filter count.

## Missing i18n keys

`ShopHome` references:
- `t('employee.allCategories')` — not in the employee namespace; exists only as `adminProducts.allCategories`
- `t('employee.noProducts')` — not defined anywhere

Both will render the raw key string as visible text. The `ProductCard` component also hardcodes `"已售罄"` instead of `t('product.soldOut')`.

## Mock session and page refresh

The mock layer keeps `currentUserId` in module scope. After login it's set correctly, but a browser refresh reloads the module and resets it to `null`. The fallback `currentUserId ?? 2` in order/points endpoints means a user who logged in as `lisi` (id=3) will see `zhangsan`'s (id=2) orders and balance after refresh. The exported `mockSetCurrentUser` function exists but is never called on app init from persisted state.

## PointsCenter sign display

`getPointsDisplay` determines the sign prefix based on `POSITIVE_TYPES` (earn, admin_add, refund -> "+", others -> "-"). The raw `tx.points` value from mock data is already signed (spend entries have negative values like `-2580`). The display renders:

```tsx
{prefix}{tx.points.toLocaleString()}
// For spend: "-" + "-2,580" = "--2,580"
```

For positive types (earn, refund, admin_add) the mock data uses positive values, so those render correctly (`+5,000`). The fix: use `Math.abs(tx.points)` in the display template, or change mock data to always use unsigned values with the type indicating direction.

## Mock order creation skips status check

`mockCreateOrder` validates stock (`product.stock <= 0`) and points balance but does not check `product.status === 1`. A delisted product (status=0) can still be redeemed through the mock if navigated to directly (e.g., `/redeem/9` for the Philips toothbrush which has `status: 0, stock: 0`). The stock check catches it incidentally in this case, but a product with `status=0, stock>0` would pass.
