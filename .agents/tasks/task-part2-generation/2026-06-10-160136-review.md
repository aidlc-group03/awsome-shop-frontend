# Full e-commerce feature implementation: services, mock, pages, and Docker (v2)

This change adds the complete PART 2 feature set for an employee points-redemption shop and its admin back-office: typed domain model, dual-mode service layer (mock/real API), six admin CRUD pages, seven employee-facing pages, shared UI components, i18n, and Docker packaging. All eight issues from the v1 review have been resolved. The build passes cleanly (835 modules, 0 errors).

Watch for: delisted products remain redeemable via direct URL navigation because neither ProductDetail nor the mock's `mockCreateOrder` checks `product.status` (confirmed); mock error messages surface raw Chinese strings to users in i18n contexts (confirmed); nav-bar points balance goes stale after redemption until full page reload (confirmed).

**Verdict**: APPROVED

## High-level view

The eight v1 issues are all properly resolved: i18n keys are defined, ProductCard uses `t()`, ShopHome passes `status: 1` server-side, App.tsx restores mock session on refresh, the 401 handler clears `auth-storage`, PointsCenter uses `Math.abs()`, ConfirmDialog/EmptyState default to i18n keys, and ApiResult detection now checks both `code` and `data` fields.

Two remaining gaps are worth noting but not blocking. The redemption flow only gates on stock and points balance — it does not check `product.status === 1`. A user who navigates directly to `/redeem/9` (a delisted product with stock > 0) can complete the purchase because `mockCreateOrder` also omits this check. ShopHome's server-side filtering prevents normal discovery of delisted products, so this is a defense-in-depth gap rather than a likely user path.

Mock error messages are Chinese-only strings (e.g., "库存不足", "积分不足"). These propagate directly to the Snackbar in Redemption's catch handler and will look broken if an English-locale user triggers a mock error.

The EmployeeLayout fetches the points balance once on mount but doesn't re-fetch after a redemption, so the nav-bar total is stale until the next navigation that remounts the layout.

## Delisted product bypass in redemption flow

ProductDetail computes `redeemDisabled` as:

```tsx
const isSoldOut = product.stock === 0;
const insufficientPoints = balance ? balance.balance < product.pointsPrice : false;
const redeemDisabled = isSoldOut || insufficientPoints;
```

A product with `status: 0` (admin-delisted) but `stock > 0` will show an enabled "Redeem Now" button. The Redemption page also loads the product by ID without checking status. Finally, `mockCreateOrder` validates stock and balance but not `product.status === 1`. Since the real backend should enforce this, the risk is limited to the mock scenario and direct-URL access.

## Mock error messages are not i18n-aware

The mock layer uses hardcoded Chinese strings for all error conditions:

- "库存不足" (out of stock)
- "积分不足" (insufficient points)
- "商品不存在" (product not found)
- "用户积分账户不存在" (account not found)

These propagate to the UI via the Redemption page's catch handler: `err instanceof Error ? err.message : t('redeem.submitFailed')`. An English-locale user who triggers a stock/balance error sees Chinese text in the Snackbar.

## Stale balance in EmployeeLayout nav-bar

The points chip in the header fetches from `pointsService.getBalance()` inside a `useEffect([], [])` — mount only. After a redemption deducts points, the nav-bar still shows the pre-redemption balance until a layout remount. A Zustand slice or custom event to signal "balance changed" after order creation would fix this.
